# 경성뎐 프로젝트 기획 포트폴리오 정리

## 프로젝트 개요

`경성뎐: 여급 실종사건`은 브라우저 기반의 스토리 ADV 프로젝트다.

이 프로젝트에서 내가 집중한 지점은 단순히 이야기를 작성하는 것이 아니라, 콘텐츠가 실제 플레이 흐름 안에서 어떻게 작동하고, 그 구조를 어떻게 관리 가능한 형태로 바꿀 것인가였다.

처음에는 빠르게 플레이 감각을 확인하는 것이 중요했기 때문에 단순한 구조로 시작했고, 이후 콘텐츠 양과 분기 복잡도가 커지면서 데이터 구조, 작업 방식, 편집 도구를 단계적으로 정리해나갔다.

즉, 이 프로젝트는 "하나의 게임을 만든 경험"인 동시에, 기획자가 콘텐츠를 더 안정적으로 설계하고 운영하기 위한 제작 구조를 직접 발전시켜온 과정이기도 하다.

---

## 1. 먼저 플레이 가능한 구조를 만들었다

프로젝트 초반에는 시스템을 복잡하게 설계하기보다, 실제 플레이 흐름이 성립하는지부터 확인하는 것이 우선이었다.

그래서 가장 먼저 한 일은 다음과 같은 기본 흐름을 갖춘 ADV 구조를 만드는 것이었다.

- 씬 전환
- 대사 진행
- 선택지 분기
- 플래그 저장
- 단서 수집
- 저장 / 불러오기

이 단계에서 중요했던 것은 "기획 문서를 예쁘게 정리하는 것"이 아니라, 플레이어가 어떤 순서로 정보를 받고 어떤 선택을 하며 어떤 결과를 만나게 되는지 실제 흐름으로 검증하는 것이었다.

기획자로서 이 단계는 아이디어를 문서에만 두지 않고, 실제 플레이 가능한 구조로 바꾸는 과정이었다.

---

## 2. 초기에는 하드코딩 방식으로 빠르게 검증했다

처음부터 데이터를 완전히 분리하지 않은 이유는, 초반에는 구조보다 속도가 더 중요했기 때문이다.

초기에는 장면, 대사, 선택지, 분기를 코드에 직접 붙여가며 빠르게 프로토타입을 만들었다.

이 방식의 장점은 분명했다.

- 씬 흐름을 빠르게 바꿔볼 수 있다
- 분기 구조를 즉시 시험할 수 있다
- 플레이 감각을 빠르게 확인할 수 있다

하지만 프로젝트가 커지면서 한계도 명확해졌다.

- 대사 수정이 곧 코드 수정이 된다
- 분기 구조를 문서처럼 한눈에 보기 어렵다
- 콘텐츠 수정과 구조 수정이 서로 얽힌다
- 기획 관점에서 전체 흐름을 관리하기가 점점 어려워진다

이때부터 "빠르게 만드는 방식"과 "지속적으로 운영 가능한 방식"은 다르다는 점이 분명해졌다.

---

## 3. 콘텐츠를 코드에서 분리해 데이터 중심 구조로 바꿨다

다음 단계에서는 콘텐츠를 코드 안에 두지 않고, 기획 데이터로 분리하는 구조 전환을 진행했다.

핵심은 다음과 같다.

- 씬 정보는 테이블로 관리
- 대사, 선택지, 분기, 단서를 각각 분리된 데이터 단위로 관리
- 런타임은 데이터를 읽어 동작하고, 콘텐츠는 별도로 수정 가능하게 구성

이 변화의 의미는 단순히 저장 형식을 바꾼 것이 아니다.

기획 관점에서 보면, 이 단계는 다음 문제를 해결하기 위한 구조 개선이었다.

- 어떤 씬이 어떤 씬으로 연결되는지 명확히 관리
- 어떤 선택지가 어떤 플래그를 남기는지 구조적으로 추적
- 조건부 대사와 엔딩 분기를 일관된 기준으로 관리
- 수정이 잦은 콘텐츠를 코드와 분리해 운영 비용을 낮춤

즉, "게임을 만드는 방식"이 개발 중심 상태에서 기획 운영이 가능한 상태로 넘어간 단계라고 볼 수 있다.

---

## 4. 엑셀 기반 작업 방식으로 정리했다

콘텐츠가 늘어나면서 가장 중요한 것은 "누가 봐도 수정 가능한 구조"를 만드는 것이었다.

그래서 시나리오와 분기 데이터는 엑셀 기반으로 정리했다.

이 방식의 장점은 다음과 같다.

- 씬, 대사, 선택지, 단서를 표 단위로 관리할 수 있다
- 분기 조건을 일관된 규칙으로 볼 수 있다
- 순서와 관계를 명확하게 통제할 수 있다
- 수정 요청이나 검토 과정이 쉬워진다

특히 기획 업무에서는 내용 그 자체만큼이나 "수정 가능성"과 "검수 가능성"이 중요하다.

엑셀 기반 구조는 단순히 익숙해서 선택한 것이 아니라, 대량의 내러티브 콘텐츠를 운영할 때 수정과 확인이 쉬운 포맷이 필요했기 때문에 선택한 작업 방식이다.

---

## 5. `xlsx -> game_data` 변환 구조를 만들었다

엑셀을 단순 참고 문서로 두지 않고, 실제 게임 데이터의 원본으로 삼기 위해 변환 파이프라인을 붙였다.

구조는 다음과 같다.

- 원본 기획 데이터는 `xlsx`
- 정해진 시트 구조를 읽어
- 런타임에서 바로 읽을 수 있는 `game_data` 형태로 변환

이 구조를 만든 이유는 명확하다.

- 기획 데이터와 실제 게임 데이터의 연결 관계를 분명히 하기 위해
- 수작업 복붙 과정에서 생기는 오류를 줄이기 위해
- 콘텐츠 수정 이후 반영 과정을 반복 가능한 방식으로 만들기 위해

기획자의 입장에서 이 단계는 중요하다.

단순히 "문서를 작성했다"가 아니라, 문서가 실제 게임에 반영되는 경로를 관리 가능한 프로세스로 만든 것이기 때문이다.

즉, 콘텐츠를 설계하는 데서 끝나지 않고, 그 설계가 안정적으로 반영되도록 작업 흐름까지 정리한 것이다.

---

## 6. 데이터 편집용 에디터를 만들었다

엑셀 분리만으로도 많이 나아졌지만, 실제로 작업해보면 또 다른 문제가 생긴다.

콘텐츠 양이 늘어나면 다음 같은 작업이 반복된다.

- 씬 연결 수정
- 대사 수정
- 선택지 수정
- 분기 조건 확인
- 단서 연결 상태 확인

이 과정을 계속 표 기반으로만 다루면 수정 비용이 다시 커진다.

그래서 프로젝트 전용 에디터를 만들었다.

이 에디터는 단순한 편의 기능이 아니라, 기획 작업 자체를 더 빠르게 만들기 위한 도구였다.

에디터를 통해 다음이 가능해졌다.

- 씬 메타 정보 수정
- 대사 / 선택지 / 단서 / 분기 구조를 한 화면에서 편집
- 현재 데이터 상태를 빠르게 검토
- 반복 수정 시 맥락 전환 비용 감소

기획 관점에서 이 단계의 핵심은 "데이터 구조를 만든 것"이 아니라, "그 데이터를 실제로 다루기 쉬운 작업 환경까지 만든 것"이다.

---

## 7. 보기 쉬운 구조를 위해 노드형 에디터로 확장했다

폼 기반 편집기만으로는 개별 항목 수정은 편하지만, 전체 흐름을 직관적으로 보기 어렵다는 문제가 남아 있었다.

특히 ADV처럼 분기 구조가 중요한 프로젝트에서는 "현재 데이터가 맞는가"만큼이나 "전체 구조가 어떻게 흘러가는가"를 빠르게 파악하는 것이 중요하다.

그래서 다음 단계로 노드형 시각 에디터를 만들었다.

이 에디터는 다음 문제를 해결하기 위한 것이었다.

- 씬 간 연결 구조를 한눈에 보기 어렵다
- 어떤 선택지가 어느 씬으로 이어지는지 추적이 번거롭다
- 분기 설계가 복잡해질수록 검토 비용이 커진다

노드형 구조를 도입하면서 다음 효과를 기대할 수 있었다.

- 씬 연결 관계를 시각적으로 파악
- 선택지와 분기 흐름을 빠르게 검토
- 전체 서사 구조 이해 비용 감소
- 수정 포인트를 더 직관적으로 확인

이 단계는 단순히 "보기 좋은 에디터"를 만든 것이 아니라, 기획자가 분기 구조를 더 잘 설계하고 검토할 수 있도록 시각적 이해 도구를 만든 것에 가깝다.

---

## 8. 이 프로젝트에서 보여주고 싶은 기획 역량

이 프로젝트에서 보여주고 싶은 것은 구현 기술 자체보다, 콘텐츠 구조를 설계하고 그 구조를 운영 가능한 형태로 발전시킨 기획 역량이다.

정리하면 다음과 같다.

- 먼저 플레이 가능한 최소 구조를 정의하고 검증했다
- 하드코딩 기반 프로토타입으로 빠르게 플레이 감각을 확인했다
- 콘텐츠와 로직을 분리해 데이터 중심 구조로 전환했다
- 엑셀 기반으로 기획 데이터 관리 체계를 만들었다
- `xlsx -> game_data` 파이프라인으로 반영 프로세스를 정리했다
- 전용 에디터를 통해 반복 수정 비용을 낮췄다
- 노드형 시각 에디터로 분기 구조의 가독성과 검토 효율을 높였다

즉, 이 프로젝트는 하나의 시나리오를 작성한 경험이 아니라, 콘텐츠를 설계하고 운영하고 확장하는 제작 구조를 스스로 정리해온 경험으로 설명할 수 있다.

---

## 한 줄 요약

초기 하드코딩 프로토타입에서 출발해, ADV 콘텐츠를 운영 가능한 데이터 구조로 분리하고, 엑셀 기반 작업 방식, 데이터 변환 파이프라인, 전용 편집기, 노드형 시각 에디터까지 단계적으로 정리해온 내러티브 콘텐츠 제작 프로젝트.

---

## 포트폴리오 소개 문장 예시

`경성뎐` 프로젝트는 브라우저 기반 ADV 게임을 직접 제작하면서, 초기 하드코딩 프로토타입에서 출발해 콘텐츠를 운영 가능한 데이터 구조로 정리하고, 엑셀 기반 작업 방식과 데이터 변환 파이프라인, 전용 편집기, 노드형 시각 에디터까지 단계적으로 확장한 프로젝트입니다. 기획자로서 단순히 콘텐츠를 작성하는 데서 그치지 않고, 콘텐츠가 실제로 관리되고 수정되고 검토되는 제작 구조까지 함께 설계한 경험을 담고 있습니다.

## 면접/발표용 설명 예시

이 프로젝트는 처음부터 큰 시스템을 한 번에 설계한 것이 아니라, 실제 제작 과정에서 생긴 문제를 하나씩 해결하며 발전시킨 작업입니다. 처음에는 하드코딩 방식으로 플레이 흐름을 빠르게 검증했고, 이후 콘텐츠 양이 늘어나면서 데이터를 코드에서 분리해 엑셀 기반으로 관리했습니다. 그 다음에는 그 데이터를 실제 게임에 안정적으로 반영하기 위해 변환 파이프라인을 만들었고, 반복 수정 비용을 줄이기 위해 전용 에디터와 노드형 시각 편집기까지 확장했습니다. 그래서 이 프로젝트는 결과물 자체보다, 콘텐츠를 구조화하고 운영 가능한 형태로 발전시킨 과정이 핵심인 작업이라고 설명할 수 있습니다.
