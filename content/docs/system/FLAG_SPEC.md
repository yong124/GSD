# 경성뎐 플래그 명세서

## 1. 문서 목적

본 문서는 `경성뎐` 프로젝트에서 사용하는 플래그의 역할과 사용 위치를 정리한 기획 문서다.

플래그는 다음 세 가지 용도로 사용된다.

- 선택 결과 저장
- 조건 대사 표시
- 후속 분기 결정

---

## 2. 운용 원칙

- 플래그는 가능한 한 `선택지`에서 기록한다.
- `Dialogue.condition`과 `Branch`는 같은 플래그 체계를 읽는다.
- 하나의 플래그는 가능한 한 하나의 의미만 가지도록 유지한다.
- 네이밍은 `flag_역할명` 형태를 기본으로 한다.

---

## 4. 기록 위치

플래그는 기본적으로 `ChoiceTable`에서 아래 컬럼으로 기록한다.

- `FlagKey`
- `FlagValue`

기록 흐름은 다음과 같다.

```text
플레이어 선택
→ Choice.FlagKey / FlagValue 저장
→ 이후 Dialogue.condition / Branch 에서 읽음
```

---

## 5. 읽는 위치

### 5-1. 조건 대사

`DialogTable`의 아래 컬럼을 사용한다.

- `ConditionKey`
- `ConditionValue`

용도:

- 특정 선택 이후에만 보이는 대사
- 이전 조사 결과에 따른 반응 차이
- 상태값에 따른 감정선 변화 표현

### 5-2. 분기

`BranchTable`의 아래 컬럼을 사용한다.

- `FlagKey`
- `FlagValue`
- `NextScene`

용도:

- 씬 종료 후 다음 씬 분기
- 엔딩 전후의 조건 분기
- 특정 루트 강제 진입

---

## 6. 작성 규칙

### 6-1. 키 규칙

- 키는 영문 소문자 + 언더스코어 기준으로 작성한다.
- 동일 의미를 다른 키명으로 중복 생성하지 않는다.
- UI 표현용 이름이 아니라, 상태값 의미가 드러나는 이름으로 작성한다.

### 6-2. 값 규칙

- 단순 양자택일은 `A / B` 또는 `yes / no`처럼 명확하게 통일한다.
- 엔딩 분기처럼 선택지가 3개 이상이면 `A / B / C` 형태를 유지한다.
- `true / false`를 사용할 경우 다른 플래그와 혼용 기준을 문서에서 명확히 한다.

### 6-3. 검수 규칙

- 같은 플래그를 서로 다른 의미로 재사용하지 않는다.
- 값 스펙이 씬마다 달라지지 않게 유지한다.
- 읽는 곳이 없는 플래그는 정리 후보로 관리한다.
- 존재하지 않는 플래그를 읽는 조건 대사는 검수 대상이다.

---

## 7. 향후 확장 방향

- 플래그를 `서사 선택`, `조사 진행`, `엔딩 상태` 등 범주별로 구분하는 체계 도입
- 플래그별 사용처 자동 목록화
- Node Editor 내 플래그 검색/참조 추적과 문서 동기화

---

## 8. 한 줄 정리

플래그는 `선택의 결과를 기록하고, 이후 대사와 분기에서 읽어 서사 흐름을 제어하는 상태값 시스템`이다.
