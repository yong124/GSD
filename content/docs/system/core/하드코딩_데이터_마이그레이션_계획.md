# 하드코딩 데이터 마이그레이션 계획

## 목적

현재 경성뎐 런타임에는 UI 편의와 빠른 구현을 위해 코드에 직접 박아둔 데이터가 남아 있다.

이 문서는 그 하드코딩 항목을 아래 세 가지로 나눠 정리한다.

1. 기존 테이블에 병합할 것
2. 새 테이블로 분리할 것
3. 런타임 계산 로직으로 유지할 것

핵심 목표는 단순히 테이블 수를 늘리는 것이 아니라, **데이터 책임을 가장 자연스러운 기존 구조에 재배치하는 것**이다.

---

## 분류 원칙

### 1. 기존 테이블 병합

이미 성격이 명확한 상위 테이블이 있으면, 새 테이블보다 기존 테이블 확장을 우선한다.

예:

- 인물 설명 → `CharacterTable`
- 단서 카테고리 → `EvidenceTable`
- 현재 목표 문구 → `SceneTable`

### 2. 새 테이블 분리

기존 테이블 어디에도 자연스럽게 속하지 않는 메타 정보는 별도 테이블로 분리한다.

예:

- 질문 목록
- 상태 단계별 UI 문구

### 3. 로직 유지

플레이 진행 상태에 따라 실시간 계산되는 정보는 억지로 테이블화하지 않는다.

예:

- 현재까지 등장한 인물 계산
- 현재 씬까지의 진행 인덱스 계산

---

## 현재 하드코딩 항목 정리

| 항목 | 현재 위치 | 성격 | 권장 이동 대상 | 방식 |
| --- | --- | --- | --- | --- |
| 인물 역할/요약 (`CHARACTER_NOTES`) | `game/js/engine/evidence.js` | 인물 메타 데이터 | `CharacterTable` | 기존 테이블 병합 |
| 질문 목록 (`getQuestionEntries`) | `game/js/engine/evidence.js` | 조사 메타 데이터 | `QuestionTable` | 새 테이블 |
| 단서 카테고리 규칙 (`getEvidenceCategory`) | `game/js/engine/evidence.js` | 단서 메타 데이터 | `EvidenceTable` | 기존 테이블 병합 |
| 상태 단계 문구 (`getResonanceState`, `getTrustState`, `getInvestigationState`) | `game/js/engine/evidence.js` | 상태 해석 데이터 | `StateDescriptorTable` | 새 테이블 |
| 현재 목표 (`goal_kicker`, `goal_text`) | `game_data.js` 씬 데이터 / 런타임 참조 | 씬 메타 데이터 | `SceneTable` | 기존 테이블 병합 |

| 등장 인물 계산 (`getRevealedCharacterIds`) | `game/js/engine/evidence.js` | 진행 기반 계산 로직 | 유지 | 로직 유지 |
| 현재 진행 순서 계산 (`getSceneProgressIndex`) | `game/js/engine/evidence.js` | 진행 기반 계산 로직 | 유지 | 로직 유지 |

## 1차 병합 대상

## 1-1. CharacterTable 확장

현재 `CharacterTable`은 이름/기본 감정/기본 이미지까지만 가진다.

여기에 수첩용 설명을 병합할 수 있다.

### 추가 권장 컬럼

- `RoleText`
- `NotebookSummary1`
- `NotebookSummary2`

### 이유

- 인물 수첩 정보는 캐릭터 고유 정보에 가깝다.
- 별도 `CharacterNotebookTable` 없이도 1차 운영이 충분하다.
- `EditorNode`와 generated xlsx에서도 가장 이해하기 쉽다.

---

## 1-2. EvidenceTable 확장

현재 단서 카테고리는 런타임에서 정규식으로 분류한다.

이 분류는 데이터로 옮기는 편이 맞다.

### 추가 권장 컬럼

- `CategoryID`
- `CategoryTitle`
- `CategoryHint`

### 이유

- 단서 분류는 단서 자체의 메타 정보다.
- 정규식 기반 분류는 유지보수성이 낮다.
- 카테고리명/힌트가 바뀔 때 코드 수정이 필요 없어야 한다.

---

## 1-3. SceneTable 확장

현재 목표 문구는 이미 씬 메타 데이터로 쓰이고 있다.

따라서 별도 목표 테이블보다 `SceneTable` 확장이 맞다.

### 추가 권장 컬럼

- `GoalKicker`
- `GoalText`

### 이유

- 목표는 씬 단위로 바뀐다.
- HUD와 조사 수첩이 같은 씬 메타를 공유할 수 있다.

---

## 2차 신설 대상

## 2-1. QuestionTable

질문은 씬, 인물, 단서 어디 하나에 완전히 종속되지 않는다.

따라서 별도 테이블이 적합하다.

### 권장 컬럼

- `QuestionID`
- `Title`
- `Detail`
- `StateLabel`
- `SortOrder`
- `Category`
- `VisibleRuleID`
- `StateRuleID`

### 1차 단순안

처음에는 아래만으로 시작해도 된다.

- `QuestionID`
- `Title`
- `Detail`
- `SortOrder`
- `VisibleRuleID`
- `StateRuleID`

### 이유

- 질문 탭은 조사 수첩의 독립 메타 구조다.
- 나중에 추리 퍼즐 진입점으로 확장하기 좋다.

---

## 2-2. StateDescriptorTable

공명/신뢰/조사 단계별 UI 문구는 현재 함수에 하드코딩돼 있다.

이건 `FlagTable`과 연결 가능한 별도 상태 해석 테이블로 가는 것이 맞다.

### 권장 컬럼

- `DescriptorID`
- `TargetFlagID`
- `MinValue`
- `MaxValue`
- `Label`
- `Detail`

### 예시

- `ResonanceLevel / 0~0 / 안정`
- `ResonanceLevel / 1~1 / 전조`
- `ResonanceLevel / 2~2 / 심화`
- `ResonanceLevel / 3~999 / 침식`

### 이유

- UI 상태 문구가 코드가 아니라 데이터가 된다.
- 수첩/상단 HUD/후속 UI가 같은 해석 규칙을 공유할 수 있다.

---

## 3차 검토 대상

## RuleTable

질문 노출, 상태 노출, 인물 조건을 일반화하려면 결국 규칙 테이블이 필요할 수 있다.

다만 1차부터 일반화하면 과설계가 될 수 있다.

따라서 권장 순서는 다음과 같다.

1. `QuestionTable`에 `VisibleRuleID`, `StateRuleID`만 먼저 둔다
2. 실제 패턴이 늘어나면 그때 `RuleTable`을 도입한다

### 권장 컬럼 예시

- `RuleID`
- `TargetType`
- `FlagID`
- `Operator`
- `Value`
- `JoinType`
- `GroupID`

---

## 런타임 로직으로 유지할 것

아래는 테이블로 옮기기보다 코드에서 계산하는 편이 자연스럽다.

### 1. 등장 인물 계산

- 현재 씬까지 실제로 `SpeakerID`가 드러났는가
- 현재 진행 줄까지 포함되는가

이건 플레이 상태 기반 계산이라 테이블보다 로직이 적합하다.

### 2. 현재 씬 진행 순서

- `State.currentSceneId`
- `State.dialogueIndex`

이 역시 순수 런타임 상태이므로 로직 유지가 맞다.

---

## 추천 마이그레이션 순서

### 1단계

- `CharacterTable` 확장
- `EvidenceTable` 확장
- `SceneTable` 목표 컬럼 명시

### 2단계

- `QuestionTable` 추가
- 조사 수첩 질문 탭을 이 테이블 기준으로 전환

### 3단계

- `StateDescriptorTable` 추가
- 상태 탭과 HUD가 같은 해석 규칙을 읽게 전환

### 4단계

- 조건식 패턴이 충분히 쌓이면 `RuleTable` 검토

---

## 결론

현재 하드코딩을 모두 새 테이블로 쪼개는 것은 좋은 방향이 아니다.

더 좋은 방향은 아래와 같다.

- 인물 설명은 `CharacterTable`로 흡수
- 단서 카테고리는 `EvidenceTable`로 흡수
- 목표 문구는 `SceneTable`로 흡수
- 질문은 `QuestionTable`로 분리
- 상태 단계 문구는 `StateDescriptorTable`로 분리
- 진행 계산은 런타임 로직으로 유지

즉, **새 테이블 신설은 최소화하고, 책임이 명확한 데이터는 기존 테이블에 병합하는 방식으로 마이그레이션한다.**
