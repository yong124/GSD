# 조사 우선순위 시스템 계획

## 1. 문서 목적

이 문서는 `경성뎐`의 게임성을 강화하기 위한 핵심 시스템으로
`조사 우선순위 시스템`을 정의하고, 다음 작업 턴에서 바로 구현할 수 있도록

- 플레이 목표
- 플레이 루프
- 데이터 구조
- UI 흐름
- 씬 적용 예시
- 단계별 구현 순서

를 한 번에 정리한 실행 문서다.

---

## 2. 왜 이 시스템이 필요한가

현재 `경성뎐`은 분위기, 문장 밀도, 연출, 캐릭터 존재감은 많이 올라왔다.
반면 플레이어가 실제로 하는 행동은 아직 `읽기`와 `선택지 클릭`에 가깝다.

그래서 지금 가장 필요한 것은
새 텍스트를 더 쓰는 것보다
`무엇을 먼저 볼지`, `누구를 먼저 캐물을지`, `어느 정보를 지금 꺼낼지`
를 플레이어가 판단하게 만드는 구조다.

이 시스템의 목표는 다음 세 가지다.

1. 플레이어를 수동적인 독자에서 능동적인 조사자로 바꾼다.
2. 기존 변수인 `InvestigationScore`, `SongsoonTrust`, `ResonanceLevel`을 플레이 감각으로 번역한다.
3. 이 프로젝트를 포트폴리오에서 `분기형 비주얼노벨`이 아니라 `조사 구조를 설계한 시스템 기획 사례`로 보이게 만든다.

---

## 3. 핵심 정의

`조사 우선순위 시스템`은
각 조사 씬에서 플레이어에게 여러 조사 대상 또는 질문 후보를 제시하되,
모든 항목을 다 깊게 볼 수 없게 제한하는 구조다.

플레이어는 제한된 기회 안에서

- 무엇을 먼저 조사할지
- 누구를 먼저 상대할지
- 지금 안전한 선택을 할지 위험한 선택을 할지

를 결정하게 된다.

그 결과에 따라

- 획득 정보
- 관계 변화
- 공명 상승 여부
- 이후 대사 반응
- 후속 분기 진입 조건

이 달라진다.

---

## 4. 플레이 루프

한 씬의 기본 루프는 아래처럼 잡는다.

1. 씬 진입
2. 조사 포인트 또는 질문 후보 노출
3. 플레이어가 1개 선택
4. 즉시 결과 반영
5. 남은 기회가 있으면 다시 선택
6. 기회를 모두 쓰거나 종료 조건 충족 시 다음 일반 대사/분기로 이동

이때 중요한 것은 `다 볼 수 없음`이다.
핵심은 항목 수가 아니라 `우선순위를 정해야 한다`는 압박이다.

---

## 5. 기대 플레이 감각

이 시스템이 붙으면 플레이어는 다음 감각을 받게 된다.

- `이번 장면에서 무엇이 더 중요한가`
- `지금 이 질문은 너무 이른가`
- `관계를 지키려면 어디까지 물어야 하나`
- `진실을 캐려면 위험을 감수해야 하나`

즉 감정선과 조사선이 분리되지 않고,
조사 선택 자체가 관계와 공포를 건드리는 방향으로 설계된다.

---

## 6. 시스템 규칙

## 6-1. 기본 규칙

- 조사 씬마다 `PriorityBudget`를 둔다.
- 플레이어는 예를 들어 3개 후보 중 2개까지만 깊게 선택할 수 있다.
- 선택 결과는 즉시 대사와 변수 변화로 반영한다.
- 선택하지 않은 항목은 이후 약화된 정보로만 회수되거나 완전히 놓칠 수 있다.

## 6-2. 선택 결과 유형

각 선택은 아래 중 하나 이상을 발생시킨다.

- 정보 획득
- 관계 변화
- 공명 변화
- 추가 질문 해금
- 특정 질문 봉쇄
- 증거 획득
- 후속 씬 대사 변화

## 6-3. 위험 선택 규칙

일부 선택은 `위험 태그`를 가진다.

예:

- 너무 직접적인 추궁
- 의식/환청/혈흔을 먼저 파고드는 선택
- 신뢰가 낮은 인물에게 증거를 바로 들이미는 선택

이런 선택은 보상이 큰 대신
`SongsoonTrust 하락`, `ResonanceLevel 상승`, `대화 조기 종료` 같은 비용을 동반한다.

---

## 7. 데이터 구조 제안

현재 구조를 크게 깨지 않고 붙이기 위해
새 테이블을 먼저 늘리기보다 `DialogTable`과 `ChoiceTable` 확장 중심으로 시작한다.

## 7-1. 최소 구현 컬럼

### DialogTable 추가 컬럼

| Column | Type | Required | Description |
| --- | --- | --- | --- |
| `PriorityGroup` | string |  | 같은 조사 묶음 ID |
| `PriorityTag` | string |  | 조사 항목 키 |
| `PriorityCost` | number |  | 이 선택이 소모하는 기회 |
| `PriorityReward` | string |  | 대표 보상 키 |
| `PriorityRisk` | string |  | 대표 위험 키 |

### ChoiceTable 추가 컬럼

| Column | Type | Required | Description |
| --- | --- | --- | --- |
| `PriorityGroup` | string |  | 어느 조사 묶음에 속하는지 |
| `PriorityTag` | string | O | 선택 항목 키 |
| `PriorityCost` | number |  | 기본 1 |
| `PriorityNeedFlag` | string |  | 노출 조건 |
| `PriorityBlockFlag` | string |  | 봉쇄 조건 |
| `PriorityResultFlag` | string |  | 선택 결과 기록 |
| `PriorityResultValue` | string |  | 선택 결과 값 |

이 단계에서는 전용 테이블 없이도 충분히 시작 가능하다.

## 7-2. 권장 런타임 상태값

- `CurrentPriorityGroup`
- `PriorityBudget_<GroupID>`
- `PrioritySpent_<GroupID>`
- `PriorityPicked_<Tag>`

예:

- `PriorityBudget_CafeTalk = 2`
- `PrioritySpent_CafeTalk = 1`
- `PriorityPicked_SongsoonFace = true`

---

## 8. UI 흐름 제안

구현 첫 버전은 복잡하게 가지 않는다.

1. 일반 대사 흐름 진행
2. 특정 지점에서 `조사 선택지 패널` 노출
3. 각 선택지는 `무엇을 조사할지` 또는 `어떤 질문을 던질지`를 의미
4. 한 개를 선택하면 결과 대사 재생
5. 예산이 남아 있으면 다시 후보 제시
6. 예산 소진 시 자동 종료

표현 방식은 아래가 적당하다.

- 상단 또는 선택지 패널에 `조사 기회 2/3` 표시
- 위험 선택은 문구 톤이나 태그로 암시
- 이미 고른 항목은 비활성화
- 봉쇄된 항목은 숨기거나 조건 부족 상태로 표시

---

## 9. 변수 연결 방향

이 시스템은 기존 변수와 자연스럽게 연결된다.

### InvestigationScore

- 현장 단서, 기록, 발자국, 문서 확인 계열 선택 보상

### SongsoonTrust

- 위로, 배려, 우회 질문, 믿어주는 선택 보상
- 몰아붙이기, 의심, 직접 추궁 계열 선택 리스크

### ResonanceLevel

- 의식 흔적, 푸른 자국, 환청, 혈흔, 금기 기록 계열 선택 리스크 또는 보상

핵심은 선택 결과가 나중 엔딩에서만 쓰이는 것이 아니라
같은 씬과 다음 씬에서 바로 반응해야 한다는 점이다.

## 10. 실제 씬 적용 예시

## 10-1. Ch2_Cafe

상황:
송순과의 대화에서 정보를 더 캐낼 수 있지만 너무 밀어붙이면 신뢰가 흔들린다.

조사 그룹:
`CafeTalk`

예산:
`2`

후보:

1. `LastMeeting`
설명: 그날 마지막으로 본 사람을 묻는다.
결과: `InvestigationScore +1`

2. `HospitalRumor`
설명: 병원 이야기를 직접 꺼낸다.
결과: 정보량 큼
리스크: `SongsoonTrust -1`

3. `ReadSongsoon`
설명: 송순의 표정과 말끝을 관찰한다.
결과: 관계 유지, 후속 감정 힌트 해금

의도:
같은 카페 씬이라도
`수사형 플레이`, `관계형 플레이`, `불온한 선택을 감수하는 플레이`
가 갈리게 만든다.

## 10-2. Ch3_Warehouse

상황:
창고 현장은 조사 포인트가 명확한 대신, 무엇을 먼저 보는가에 따라 정보 결이 달라진다.

조사 그룹:
`WarehouseSweep`

예산:
`2`

후보:

1. `Footprints`
결과: `InvestigationScore +1`

2. `CrateMark`
결과: 조직/유통 관련 단서 확보

3. `FloorTrace`
결과: `ResonanceLevel +1`, 의식 흔적 파악

의도:
플레이어가 `사실 추적`과 `공포 추적` 중 무엇을 우선하는지 보이게 한다.

## 10-3. Ch5_Ritual_Room

상황:
의식실은 정보량보다 압박과 공명이 중요한 구간이다.

조사 그룹:
`RitualRoom`

예산:
`2`

후보:

1. `Ledger`
결과: 기록 확보, 후속 엔딩 힌트 강화

2. `Mask`
결과: 공포 연출, `ResonanceLevel +1`

3. `LookAtSongsoon`
결과: 관계 반응, 감정 대사 변화

의도:
무엇을 먼저 보는가가 단순한 정보량이 아니라
`기록 집착`, `금기 추적`, `관계 우선`
중 어느 플레이인지 보여주게 한다.

## 11. 구현 우선순위

## 11-1. 1단계

- `ChoiceTable`에 우선순위 관련 컬럼 추가
- 런타임에 `PriorityBudget`과 `PrioritySpent` 처리
- 한 씬에서 1개 그룹만 동작하게 제한

목표:
최소 기능 검증

## 11-2. 2단계

- `Ch2_Cafe`, `Ch3_Warehouse`, `Ch5_Ritual_Room` 3개 씬만 시범 적용
- 결과 대사와 변수 반영 연결
- 선택 후 즉시 피드백 텍스트 보강

목표:
실제 게임성이 올라가는지 확인

## 11-3. 3단계

- EditorNode에서 `PriorityGroup`, `PriorityTag`, `PriorityCost` 편집 가능하게 추가
- generated xlsx와 validate 스크립트 반영

목표:
운영 가능성 확보

## 11-4. 4단계

- 증거 제시와 조사 우선순위 시스템 연결
- 위험 선택 연출과 스탠딩/FX 연동
- 봉쇄/해금 피드백 강화

목표:
포트폴리오 수준의 완성도 확보

---

## 12. 내일 바로 할 일

내일 시작 순서는 아래가 가장 안전하다.

1. `ChoiceTable` 확장 컬럼 확정
2. 런타임에서 `PriorityBudget` 처리
3. `Ch2_Cafe` 시범 적용
4. 플레이 감각 확인
5. 괜찮으면 `Ch3_Warehouse`, `Ch5_Ritual_Room` 확장

첫날 목표는 `전 구간 적용`이 아니라
`한 씬에서 진짜 플레이 감각이 생기는지 확인`이다.

---

## 13. 한 줄 정리

`조사 우선순위 시스템`은 경성뎐을 단순한 선택형 비주얼노벨에서
`무엇을 먼저 파고들지 판단하게 만드는 조사 게임`으로 밀어주는 첫 번째 핵심 시스템이다.
