# 새 테이블 구조 전환 계획

## 목적

`테이블 완전 새구조.md`를 기준으로, 현재 경성뎐의 데이터/에디터/파이프라인/런타임을 새 구조에 맞게 재정렬한다.

이번 라운드에서는 다음 원칙을 고정한다.

- `Question / Rule / StateDescriptor`는 당분간 런타임 하드코딩 유지
- 나머지 데이터 구조는 새 테이블 구조로 전환
- 문서 -> 파이프라인 -> 에디터 -> 런타임 -> 데이터 마이그레이션 순서로 진행

## 현재 런타임 필드 ↔ 새 구조 대응표

| 현재 구조 | 새 구조 | 비고 |
| --- | --- | --- |
| `Scene.next_scene` | `BranchTable.NextSceneID` | Scene 종료 후 기본 이동도 Branch 평가 흐름으로 통일 예정 |
| `Scene.goal_kicker` | `SceneTable.GoalKicker` | 유지 |
| `Scene.goal_text` | `SceneTable.GoalText` | 유지 |
| `Scene.evidence_prompt_title` | `SceneTable.EvidencePromptTitle` | 유지 |
| `Scene.evidence_prompt_hint` | `SceneTable.EvidencePromptHint` | 유지 |
| `Scene.priority_title` | `InvestigationTable.Title` | 조사 메타로 이동 |
| `Scene.priority_hint` | `InvestigationTable.Hint` | 조사 메타로 이동 |
| `Scene.priority_budget` | `InvestigationTable.Budget` | 조사 메타로 이동 |
| `Scene.priority_dialogues` | `DialogTable + ChoiceGroupTable` | 조사 선택 결과 흐름으로 재구성 |
| `Scene.priority_after_dialogues` | `DialogTable.NextDialogID` 또는 전용 Dialog 흐름 | 조사 종료 후 후속 대사로 흡수 |
| `Dialog.label` | `DialogTable.DialogID` 또는 `DialogTable.NextDialogID` | 라벨 점프를 명시적 ID 연결로 전환 |
| `Dialog.condition` | `DialogTable.ConditionGroupID` | 조건 공통 테이블로 이동 |
| `Choice.flag_key / flag_value` | `ChoiceTable.Trust* / ResonanceValue / EvidenceID` + 런타임 선택 효과 | 이번 라운드에서는 새 구조 컬럼 우선으로 정리 |
| `Choice.next_scene` | `ChoiceTable.NextType=Scene`, `NextID=SceneID` | 명시적 타입 기반 이동 |
| `Choice.next_dialogue` | `ChoiceTable.NextType=Dialog`, `NextID=DialogID` | 명시적 타입 기반 이동 |
| `Choice.condition` | `ChoiceTable.ConditionGroupID` | 선택지 노출 조건 통일 |
| `Choice.priority_cost` | `ChoiceGroupTable.MaxSelectable` + `InvestigationTable.Budget` | 조사 선택 개념으로 흡수 |
| `Choice.evidence_id` | `ChoiceTable.EvidenceID` | 유지 |
| `Choice.extra_flags` | 보류 | 새 구조 문서에는 아직 직접 대응 컬럼 없음. 이번 라운드에서 최소 fallback 유지 검토 |
| `Branch.flag_key / flag_value` | `BranchTable.ConditionGroupID` | 분기 조건 공통 테이블로 이동 |
| `Evidence.category_*` | `EvidenceCategoryTable` | 분리 |

## 새 구조로 옮긴 뒤 목표 상태

- Scene는 장면 메타만 가진다
- Dialog는 대사/흐름 노드 역할만 가진다
- Choice는 그룹에 속한 개별 행동으로 동작한다
- Condition은 모든 노출/분기 조건을 공통 판정한다
- Investigation은 조사 UI 메타와 선택지 그룹 호출을 담당한다
- Evidence는 정적 정의만 가진다
- 증거 보유 여부는 런타임 상태가 관리한다
- 질문/룰/상태설명은 이번 라운드에서는 기존 방식 유지
