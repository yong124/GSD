# 현재 운영 워크플로

## 한 줄 요약

현재 GSD 운영 기준은 `game_data.js 중심 수정 -> validate -> generated xlsx -> scene-local QA`다.

## 현재 소스 오브 트루스

- 실무 수정 기준: `G:\GSD\game\data\game_data.js`
- generated 검수 산출물: `G:\GSD\content\generated\script.generated.xlsx`
- 역방향 입력 원본: `G:\GSD\content\data\script.xlsx`

즉, 평소엔 `game_data.js`를 기준으로 움직이고, 필요할 때만 `script.xlsx`를 다시 밀어 넣는다.

## 작업 흐름

### 1. 데이터 수정

- `game_data.js` 수정
- 필요하면 관련 런타임 파일 수정

### 2. 구조 검증

```powershell
py G:\GSD\content\tools\validate_game_data.py
```

### 3. generated xlsx 생성

```powershell
cmd /c G:\GSD\content\tools\run_data_to_generated_xlsx.bat
```

### 4. 브라우저 QA

- 서버 실행
- scene-local Playwright 러너로 대상 씬 확인

## 배치 파일 구분

### `run_data_to_generated_xlsx.bat`

목적:

- 현재 `game_data.js`를 검증하고
- `script.generated.xlsx`만 다시 만든다

쓰는 때:

- 데이터 수정 후 generated 검수 파일만 갱신할 때

### `run_export.bat`

목적:

- `script.xlsx -> game_data.js`
- 이후 `script.generated.xlsx`까지 다시 만든다

쓰는 때:

- 엑셀 원본을 다시 게임 데이터로 반영해야 할 때

## 현재 QA 기준

- 브라우저 QA는 `.qa-node` 러너 우선
- 전체 풀런보다 대상 씬 개별 확인 우선
- 조건부 분기는 `QA_GAUGES`, `QA_FACTS`, `QA_CHOICES`로 상태 주입 후 확인

## 유지 원칙

- 과거 전환 계획 문서는 현재 기준 문서로 쓰지 않는다.
- 실제로 지금 실행되는 경로만 남긴다.
- README와 QA 가이드는 현재 워크플로와 다르면 즉시 갱신한다.
