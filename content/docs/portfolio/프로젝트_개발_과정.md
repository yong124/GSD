# 경성뎐 프로젝트 발전 과정

## 1. 문서 목적

본 문서는 `경성뎐` 프로젝트가 어떤 문제를 해결하면서 현재 구조까지 발전했는지 정리한 포트폴리오용 기록 문서다.

핵심은 단순히 결과물을 나열하는 것이 아니라,

- 어떤 한계가 있었는지
- 왜 다음 단계의 구조가 필요했는지
- 기획자로서 어떤 판단을 했는지

를 보여주는 데 있다.

---

## 2. 프로젝트 출발점

프로젝트의 시작점은 `1930년대 경성`, `여급 실종`, `사이비 종교`, `기록과 삭제`라는 테마를 가진 내러티브 게임 기획이었다.

초기에는 먼저 플레이 가능한 형태를 만드는 것이 중요했기 때문에,

- 장면 전환
- 대사 진행
- 선택지
- 간단한 분기

를 처리할 수 있는 최소한의 ADV 엔진부터 직접 구성했다.

이 단계의 목표는 완성도가 아니라,
`이 소재가 실제 인터랙티브 서사로 작동하는지`를 빠르게 검증하는 것이었다.

---

## 3. 1단계: 하드코딩 기반 프로토타입

초기 버전은 장면과 흐름을 코드 안에 직접 작성하는 방식으로 만들었다.

이 방식의 장점은 빠르다는 것이었다.
기획 변경이 생기면 바로 코드에서 문장을 바꾸고 흐름을 고칠 수 있었다.

하지만 곧 한계가 분명해졌다.

- 장면 수가 늘어날수록 수정 위치를 찾기 어렵다
- 분기 구조를 한눈에 보기 어렵다
- 기획 검수와 데이터 검수가 코드 의존적이다
- 스토리 수정이 곧 코드 수정이 된다

즉, `프로토타입 검증` 단계로는 적합했지만,
콘텐츠 운영 단계로 넘어가기엔 구조가 불안정했다.

---

## 4. 2단계: 데이터 분리와 xlsx 파이프라인

콘텐츠 규모가 커지면서, 시나리오를 코드에서 분리할 필요가 생겼다.

그래서 `script.xlsx`를 중심으로

- Scene
- Dialogue
- Choice
- Branch
- Evidence

테이블을 관리하고,
이를 `export_to_json.py`로 변환해 `game_data.js`를 생성하는 구조를 만들었다.

이 단계에서 얻은 효과는 분명했다.

- 기획 데이터와 런타임 코드를 분리할 수 있었다
- 표 기반으로 시나리오를 정리하고 검수할 수 있었다
- 데이터 구조를 문서화하기 쉬워졌다

반면 새로운 한계도 드러났다.

- 씬 간 연결 구조를 표만으로 보기는 어렵다
- 분기와 후속 흐름을 시각적으로 추적하기 힘들다
- 대량 수정에는 강하지만 구조 설계에는 불편하다

즉, xlsx는 좋은 관리 도구였지만,
복잡한 분기형 서사를 설계하는 메인 인터페이스로는 부족했다.

---

## 5. 3단계: 에디터 제작

이 한계를 해결하기 위해 자체 편집 도구를 만들기 시작했다.

초기에는 카드형 편집 UI를 실험했다.
장면 단위로 데이터를 열고 수정하는 방식은 표보다 직관적이었지만,
프로젝트가 커질수록 다시 비슷한 문제가 드러났다.

- 전체 구조를 한눈에 보기 어렵다
- 분기 연결을 추적하기 힘들다
- 장면 사이 관계보다 개별 입력 폼 중심으로 작업하게 된다

이 실험은 실패라기보다,
`어떤 툴이 실제 기획 작업에 맞지 않는지`를 확인한 단계였다.

---

## 6. 4단계: 노드형 에디터로 전환

이후 메인 작업 툴을 `Node Editor`로 전환했다.

노드형 에디터의 목표는 단순한 편집기가 아니라,

- 씬 구조를 한눈에 보고
- 분기 흐름을 시각적으로 검수하고
- 대사 / 선택지 / 단서 / 분기를 한 자리에서 수정하는

기획 중심 제작 도구를 만드는 것이었다.

이 단계에서 프로젝트는 단순 게임 제작을 넘어,
`서사를 운영하는 툴을 직접 설계하는 단계`로 확장됐다.

실제로 Node Editor에는 이후 다음 기능이 추가되었다.

- 검색 / 필터
- 분기 및 참조 검수
- 프리뷰
- 미니맵과 시각적 레이아웃
- generated xlsx 출력 보조

즉, 에디터는 더 이상 부수 도구가 아니라
프로젝트의 메인 작업 환경이 되었다.

---

## 7. 5단계: 양방향 운영 구조

Node Editor가 메인 툴이 된 이후에는,
`xlsx -> game_data` 단방향 구조만으로는 부족해졌다.

그래서 현재는 다음과 같은 운영 개념으로 정리되어 있다.

- 메인 편집: Node Editor, 필요 시 `game_data.js` 직접 수정
- 런타임 반영: game_data
- 대량 수정 / 검수 / 공유: generated xlsx 및 script.xlsx

즉,

`Node Editor / direct edit -> game_data -> generated xlsx`

흐름을 중심으로,
필요할 때 표 기반 작업을 다시 수행할 수 있는 보조 파이프라인을 두었다.

이 구조는 기획 툴과 문서 포맷을 분리하지 않고,
상황에 따라 오갈 수 있게 만든 점이 특징이다.

이후 운영 단계에서는 여기에 한 가지가 더 붙었다.

- `validate_game_data.py`로 구조 오류를 먼저 검수

즉 현재 실무 흐름은 아래에 가깝다.

1. Node Editor 또는 `game_data.js`에서 수정
2. `validate_game_data.py`로 참조 / order / id 오류 검수
3. `json_to_generated_xlsx.py --with-delimited`로 generated xlsx와 csv/tsv 산출
4. 필요 시 `script.xlsx` 원본 표에 수동 복붙

---

## 8. 6단계: 세계관 / 캐릭터 / 관계 설계 강화

프로젝트가 구조적으로 안정된 이후에는
콘텐츠의 밀도 자체를 높이는 작업으로 넘어갔다.

이 단계에서 중점적으로 정리한 것은 아래와 같다.

- 1930년대 경성과 식민지 현실의 세계관 재정리
- 청의교와 청의동자, 기록과 삭제의 모티프 정리
- 주요 캐릭터 아크 설계
- 캐릭터 간 관계 아크 설계
- 시대극 톤과 대사 기준 정리

이를 통해 프로젝트는 단순한 사건 해결형 스토리에서,
`기록되는 사람과 지워지는 사람`,
`구원과 착취`,
`냉소에서 책임으로 이동하는 시선`
을 다루는 구조로 더 선명해졌다.

---

## 9. 현재 구조의 의미

현재 `경성뎐`은 단순히 ADV 한 편을 만든 프로젝트가 아니다.

이 프로젝트는 다음을 한 흐름으로 보여준다.

1. 내러티브 엔진 설계
2. 하드코딩 프로토타입 검증
3. 데이터 테이블 분리
4. xlsx 기반 운영 구조 설계
5. 자체 편집기 실험
6. 노드형 에디터 전환
7. 양방향 데이터 파이프라인 정리
8. 구조 검수 스크립트 추가
9. 세계관 / 캐릭터 / 관계 문서화

즉, 기획자가
`콘텐츠를 설계하는 것`에서 멈추지 않고,
`콘텐츠를 운영할 수 있는 구조와 도구까지 함께 만든 프로젝트`
라고 정리할 수 있다.

---

## 10. 포트폴리오에서 강조할 포인트

### 10-1. 시스템 기획 관점

- 내러티브 구조를 테이블과 상태값으로 설계
- 선택과 분기, 단서 수집, 상태값 누적 구조를 직접 정의
- 운영 가능한 데이터 구조와 검수 체계를 만들었다

### 10-2. 전투 / 시스템 기획 관점

- 전투 대신 `선택`, `기록`, `조사`, `감응`, `단서`를 플레이 감각의 축으로 삼았다
- 상태값이 서사와 엔딩 분기에 영향을 주는 구조를 설계했다
- 툴과 데이터 운영 구조까지 포함해 제작 파이프라인을 기획했다

### 10-3. 프로젝트 운영 관점

- 초기 구조의 한계를 인지하고 다음 단계 구조로 전환했다
- xlsx, 런타임 데이터, 에디터의 역할을 분리했다
- 문서, 데이터, 툴을 동시에 정리하며 프로젝트를 확장했다

---

## 11. 한 줄 요약

`경성뎐`은 1930년대 경성을 배경으로 한 서사형 ADV를 기획하며, 하드코딩 프로토타입에서 시작해 데이터 테이블 구조, xlsx 파이프라인, 노드형 에디터, 구조 검수 스크립트, 세계관 및 캐릭터 문서 체계까지 직접 확장해 온 프로젝트다.
