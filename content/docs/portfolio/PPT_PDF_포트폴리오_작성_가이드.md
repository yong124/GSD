# PDF 제출용 PPT 포트폴리오 작성 가이드

## 1. 문서 목적

이 문서는 현재 프로젝트를 바탕으로 게임기획자 포트폴리오를 `PPT로 제작하고 PDF로 제출`할 때의 기준을 정리한 가이드다.

중요한 전제는 하나다.

이 포트폴리오는 `발표 자료`가 아니라 `읽는 문서형 포트폴리오`에 가깝다.

즉 이 문서는 아래 방향을 기준으로 삼는다.

- 발표자가 말로 보완하는 슬라이드가 아니다
- 슬라이드만 읽어도 이해되는 문서여야 한다
- 텍스트는 필요하지만, 글만 많은 문서는 아니다
- 시각 자료를 많이 써서 읽기 부담을 줄여야 한다
- 기획 의도, 구조, 판단, 개선 과정을 보여줘야 한다

한 줄로 정리하면 아래와 같다.

`문서형 포트폴리오를 PPT 형식으로 시각화한 결과물`

---

## 2. 이 포트폴리오를 어떻게 보여줄 것인가

이 프로젝트는 아래처럼 보이면 약해진다.

- 게임 하나를 만든 사람
- AI로 빠르게 구현한 사람
- 단순히 툴을 붙인 사람

반대로 아래처럼 보여야 강하다.

- 콘텐츠 제작 구조를 설계한 게임기획자
- 시스템 개념을 데이터 구조로 정리한 사람
- 에디터와 검증 흐름까지 연결한 사람
- 제작 과정 전체를 개선한 사람

따라서 포트폴리오 전체 메시지는 아래처럼 잡는 것이 좋다.

`게임 결과물 하나를 소개하는 것이 아니라, 게임 콘텐츠를 반복 생산하고 수정하고 검증할 수 있는 구조를 설계한 프로젝트`

---

## 3. 발표용 PPT와 다른 점

### 발표용 PPT

- 텍스트가 짧다
- 키워드 위주다
- 발표자가 말로 설명해 준다
- 한 장에서 생략되는 정보가 많다

### 문서형 PPT 포트폴리오

- 슬라이드만 읽어도 이해돼야 한다
- 텍스트가 어느 정도 들어가야 한다
- 하지만 문단이 길어지면 안 된다
- 시각 자료가 설명을 대신해 줘야 한다

즉 이번 포트폴리오는 `짧게 말하는 슬라이드`가 아니라 `읽히는 문서를 예쁘게 편집한 PPT`라고 생각하는 것이 맞다.

---

## 4. 텍스트와 시각 자료의 비율

추천 비율은 아래와 같다.

- 시각 자료 60~70%
- 텍스트 30~40%

이때 텍스트는 길게 이어 쓰지 말고 아래처럼 나누는 편이 좋다.

- 제목 1줄
- 핵심 설명 2~4줄
- 보조 설명 2~3개
- 하단 요약 박스 1개

시각 자료는 아래 종류를 적극적으로 쓰는 것이 좋다.

- 에디터 캡처
- 데이터 패널 캡처
- Before / After 비교
- 시스템 구조도
- 데이터 흐름도
- 작업 파이프라인 다이어그램
- 실제 게임 화면

게임 화면은 필요하지만, 이 포트폴리오의 메인 증거는 에디터와 구조 화면이다.

---

## 5. 문서형 페이지 기본 구조

문서형 PPT는 페이지마다 아래 구조를 반복하면 안정적으로 나온다.

### 기본 구조 A

- 상단: 페이지 제목
- 좌측: 설명 문단 3~4줄
- 우측: 메인 시각 자료 1개
- 하단: 요약 박스 또는 핵심 인사이트 1개

### 기본 구조 B

- 상단: 제목
- 중앙: 큰 시각 자료
- 하단 좌측: 구조 설명 2~3줄
- 하단 우측: 포인트 3개

### 기본 구조 C

- 상단: 제목
- 좌우 분할: Before / After 비교
- 하단: 무엇이 어떻게 바뀌었는지 정리

중요한 것은 페이지마다 `읽는 흐름`이 보여야 한다는 점이다.

---

## 6. 문서형 포트폴리오에 어울리는 목차

추천 목차는 아래와 같다.

1. 표지
2. 프로젝트 개요
3. 문제 정의
4. 기획 목표
5. 시스템 구조
6. 데이터 구조 정리
7. 에디터 설계
8. 작업 흐름과 검증 방식
9. Before / After
10. 결과와 개선점
11. 회고
12. 내가 보여주고 싶은 역량

이 구조는 발표용보다 텍스트 밀도가 약간 높고, 회고와 판단이 더 분명히 드러난다.

---

## 7. 페이지에 실제로 넣어야 하는 내용

각 페이지에는 아래 네 가지 중 최소 세 가지가 들어가야 한다.

- 문제
- 판단
- 구조
- 증거

예를 들면 아래처럼 구성할 수 있다.

### 문제 정의 페이지

- 설명: 왜 기존 방식이 불편했는가
- 시각 자료: 초기 구조 또는 하드코딩 예시
- 요약 박스: 문제는 기능이 아니라 운영 구조의 부재였다는 점

### 에디터 설계 페이지

- 설명: 왜 노드 에디터가 필요했는가
- 시각 자료: 실제 에디터 캡처
- 보조 포인트: 어떤 기능이 작업 속도를 올렸는가

### 회고 페이지

- 설명: 잘한 판단, 아쉬운 판단, 다음 보완점
- 시각 자료: 변화 타임라인 또는 Before / After
- 요약 박스: 이번 프로젝트의 핵심 자산은 무엇이었는가

---

## 8. 문장 톤 기준

문서형 포트폴리오에서는 문장 톤도 중요하다.

좋은 문장 방향은 아래와 같다.

- 왜 그렇게 설계했는지 설명한다
- 구조적 판단을 드러낸다
- 결과보다 과정과 개선 이유를 말한다
- 기획자의 시선을 유지한다

예를 들어 아래 문장은 좋다.

`콘텐츠 양이 늘어날수록 필요한 것은 더 많은 하드코딩이 아니라, 구조를 한눈에 볼 수 있는 편집 도구였다.`

`이 프로젝트에서는 기획 개념이 실제 데이터 구조에 그대로 이어지도록 Condition 중심 구조로 재정리했다.`

`AI는 결과물을 대신 만든 것이 아니라 반복 구현 속도를 높여 기획 검증 시간을 확보하는 도구로 사용했다.`

반대로 아래 같은 표현은 피하는 것이 좋다.

- 열심히 만들었다
- 많은 기능을 구현했다
- 다양한 기술을 사용했다
- AI를 활용해 빠르게 만들었다

이런 문장은 기획자의 판단보다 작업량만 남는다.

---

## 9. 디자인 기준

디자인은 화려함보다 `읽기 쉬운 구조`가 우선이다.

### 추천 원칙

- 한 페이지에 메시지는 하나만 둔다
- 텍스트 블록은 3~5줄 단위로 끊는다
- 시각 자료를 크게 쓴다
- 표와 다이어그램을 적극적으로 쓴다
- 동일한 페이지 그리드를 반복해 안정감을 준다

### 추천 요소

- 상단 제목 바
- 얇은 구분선
- 박스형 요약 영역
- 강조 키워드 태그
- Before / After 비교용 카드

### 피해야 할 것

- 한 페이지에 문단을 너무 많이 넣기
- 캡처 이미지만 많이 붙이고 설명이 없는 구성
- 반대로 글만 잔뜩 넣고 PPT 느낌을 잃는 구성
- 모든 페이지 레이아웃이 제각각인 구성

## 10. 이미지와 화면 캡처 활용법

시각 자료는 장식이 아니라 증거다.

따라서 아래 우선순위로 쓰는 것이 좋다.

1. 노드 에디터 전체 화면
2. 데이터 패널 화면
3. 조건 / 상태 / 게이지 구조 캡처
4. 이미지 미리보기, 검증, 편의 기능 화면
5. 실제 게임 화면

이 포트폴리오에서 게임 화면은 결과를 보여주는 장면이고, 에디터와 구조 화면은 역량을 보여주는 장면이다.

---

## 11. PDF 제출을 위한 실무 팁

- 글자 크기는 너무 작아지지 않게 유지한다
- 캡처 이미지는 원본 해상도를 확인한다
- 흰 배경 기준으로 인쇄해도 읽히는지 확인한다
- 페이지 번호를 작게 넣어도 좋다
- 마지막에는 PDF로 내보낸 뒤 실제 읽기 흐름을 다시 확인한다

특히 PDF는 발표 상황이 아니므로, `설명 없이 읽어도 이해되는가`를 끝까지 점검해야 한다.

---

## 12. 이 프로젝트에서 특히 강조해야 하는 것

이 프로젝트는 아래 세 가지가 특히 강점이다.

- 시스템 개념을 데이터 구조로 정리한 점
- 에디터를 직접 설계해 작업 흐름을 개선한 점
- 제작과 검증 파이프라인까지 확장한 점

따라서 포트폴리오도 이 세 가지를 중심으로 읽히게 만들어야 한다.
