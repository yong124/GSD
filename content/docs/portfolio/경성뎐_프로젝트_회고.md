# 경성뎐 프로젝트 회고

## 1. 문서 목적

이 문서는 `경성뎐` 프로젝트를 처음 기획한 시점부터 현재 구조에 이르기까지,  
무엇을 만들었고 왜 그렇게 바꾸었는지 한 흐름으로 정리한 포트폴리오용 회고 문서다.

핵심은 단순히 기능 목록을 나열하는 것이 아니라,

- 어떤 문제를 먼저 풀었는지
- 어떤 한계 때문에 구조를 갈아엎었는지
- 시스템 기획자로서 어떤 판단을 했는지
- AI를 어디에 실무적으로 활용했는지

를 보여주는 데 있다.

---

## 2. 프로젝트 출발점

`경성뎐`은 1930년대 경성을 배경으로,  
실종 사건, 사이비 의식, 기록과 삭제, 식민지 조선의 현실을 묶어낸 조사형 서사 게임으로 출발했다.

초기 기획의 핵심 테마는 아래와 같았다.

- 실종과 은폐
- 기록되는 사람과 지워지는 사람
- 구원으로 포장된 착취
- 냉소에서 책임으로 이동하는 시선

처음부터 목표가 분명했던 것은,  
이 프로젝트를 단순한 분위기형 비주얼노벨로 끝내지 않는 것이었다.

즉 플레이어가 단순히 텍스트를 읽는 것이 아니라,

- 질문을 붙들고
- 단서를 모으고
- 관계와 위험을 관리하고
- 사건의 진실에 더 가까이 다가가는

구조를 만들고 싶었다.

---

## 3. 1단계: 하드코딩 프로토타입

초기 버전은 가장 빠르게 playable한 형태를 만드는 데 집중했다.

이 시기에는 장면과 흐름을 코드에 직접 적어 넣는 방식으로 진행했다.

핵심 목표는 단 하나였다.

`이 소재가 인터랙티브 서사로 실제 작동하는가`

이 단계에서 얻은 것은 분명했다.

- 세계관 톤 검증
- 주요 캐릭터 관계 초안 검증
- 기본 ADV 흐름 검증
- 사건 전개 리듬 검증

하지만 장면 수가 늘어나자 한계가 빠르게 드러났다.

- 스토리 수정이 곧 코드 수정이 된다
- 분기 구조를 한눈에 보기 어렵다
- 어떤 상태가 어디서 바뀌는지 추적하기 어렵다
- 기획 검수와 데이터 검수가 코드 의존적이다

즉, 프로토타입 단계에서는 빠르지만 운영 구조로는 불안정했다.

---

## 4. 2단계: 데이터 분리와 xlsx 파이프라인

콘텐츠 규모가 커지면서 시나리오를 코드에서 분리할 필요가 생겼다.

그래서 `script.xlsx`를 중심으로

- `Scene`
- `Dialog`
- `Choice`
- `Branch`
- `Evidence`

를 관리하고, 이를 `game_data.js`로 변환하는 파이프라인을 만들었다.

이 시기부터 프로젝트는
`게임 코드`와 `콘텐츠 데이터`를 분리해서 다루기 시작했다.

좋았던 점:

- 기획 데이터를 표로 정리하고 검수할 수 있었다
- 구조 문서화가 쉬워졌다
- 런타임과 콘텐츠 수정 책임을 나눌 수 있었다

하지만 새로운 한계도 있었다.

- 씬 간 연결 구조를 표만으로 보는 것은 여전히 답답했다
- 분기와 후속 흐름을 직관적으로 추적하기 어려웠다
- 표 기반 입력은 관리엔 좋지만 구조 설계엔 불편했다

즉, xlsx는 좋은 운영 포맷이었지만 메인 제작 도구는 아니었다.

---

## 5. 3단계: 자체 편집 도구 실험

이후 표만으로 구조를 다루는 한계를 줄이기 위해 자체 편집기를 만들기 시작했다.

처음에는 카드형 편집 UI를 시도했다.

장면 단위로 데이터를 열고 수정하는 방식은 표보다 직관적이었지만,
프로젝트가 커질수록 문제가 다시 드러났다.

- 전체 씬 연결 구조를 한눈에 보기 어렵다
- 분기 흐름을 시각적으로 검수하기 어렵다
- 장면 간 관계보다 입력 폼 위주 작업이 된다

이 실험을 통해 분명해진 것은,
`경성뎐`에는 단순 폼 기반 편집기보다 `구조를 시각화하는 에디터`가 필요하다는 점이었다.

---

## 6. 4단계: Node Editor 전환

그래서 메인 툴을 `Node Editor` 방향으로 전환했다.

목표는 단순 편집기가 아니라,

- 씬 구조를 한눈에 보고
- 파란 기본 진행선과 분기선을 시각적으로 검수하고
- 대사/선택지/단서/분기를 한 자리에서 다룰 수 있는

기획 중심 제작 도구를 만드는 것이었다.

이 단계에서 프로젝트는 단순 게임 제작을 넘어,
`서사를 운영하는 툴을 직접 설계하는 단계`로 확장됐다.

Node Editor에 이후 들어간 것:

- 검색 / 필터
- 씬 연결 시각화
- 미니맵
- 프리뷰
- 구조 검수
- generated xlsx 출력 보조
- 데이터 편집 탭 분리

즉, 에디터는 부수 도구가 아니라 프로젝트의 메인 작업 환경이 되었다.

---

## 7. 5단계: 런타임 연출 강화

기본 흐름과 데이터 구조가 어느 정도 잡힌 뒤에는,
게임이 너무 “텍스트만 읽는 화면”처럼 보이지 않도록 런타임 연출을 강화했다.

이 단계에서 추가한 것:

- `SpeakerID`, `EmotionType`
- `StandingSlot`, `FocusType`
- `EnterMotion`, `IdleMotion`, `FxType`
- 스탠딩 레이어
- 화자 강조 / 비화자 dim
- 의식실 / 창고 / 문턱 같은 핵심 장면 연출 강화

이 변화로 프로젝트는
단순한 대사 출력기에서, `장면 연출을 갖춘 조사형 서사 게임`으로 한 단계 올라갔다.

---

## 8. 6단계: 조사 시스템과 UI 확장

연출이 올라온 뒤에는 플레이어가 실제로 “무엇을 붙들고 있는지”가 보이도록 만들 필요가 있었다.

그래서 아래 시스템과 UI를 확장했다.

- 조사 HUD
- 현재 목표 카드
- 메모장 / 조사 수첩
- 상태 / 인물 / 단서 / 질문 탭
- 3슬롯 저장/불러오기
- 신규 단서 배지
- 선택 결과 토스트

이 단계의 핵심은,
정보가 단순히 쌓이는 것이 아니라 `보이는 게임성`으로 바뀌도록 만드는 것이었다.

---

## 9. 7단계: 추리 루프 추가

이후 프로젝트를 진짜 조사 게임으로 보이게 만들기 위해,
핵심 추리 루프를 실제 플레이 구조로 넣었다.

추가한 루프는 아래와 같다.

- 질문 해결
- 단서 연결
- 증거 제시
- 모순 판별

그리고 이것이 수첩 안에서만 끝나지 않도록,

- 후반 분기
- 엔딩 경사
- 에필로그 회수

까지 연결했다.

즉 플레이어가 무엇을 보고, 무엇을 묻고, 무엇을 반박했는지가  
단순한 UI 수집 요소가 아니라 실제 결과로 돌아오게 만들었다.

이 단계부터 `경성뎐`은
단순한 선택형 서사가 아니라,
`질문과 근거를 기반으로 전개가 달라지는 조사형 추리 게임`
으로 설명할 수 있게 되었다.

---

## 10. 8단계: 조건 중심 데이터 구조 재정리

프로젝트가 커지면서 기존의 `flag_key / flag_value` 구조는 유지보수 비용이 커졌다.

그래서 최근에는 데이터 구조를 다시 정리했다.

핵심 방향은 아래와 같았다.

- 조건은 `ConditionGroup` 중심으로 통일
- 선택 이력은 `ChoiceSelected`로 읽기
- 증거 보유는 `EvidenceOwned`로 읽기
- 신뢰도는 `Trust(CharacterID)`로 읽기
- 공명도, 조사 점수, 의식 이해도는 `StateValue`로 정리
- `NextScene / NextDialogue`는 `NextType / NextID`로 통일

이 과정에서 실제로 레거시를 걷어냈다.

- 대사 조건 레거시 제거
- 브랜치의 `flag_key / flag_value` 제거
- 선택지의 `flag_key / extra_flags` 제거
- `ChoiceSelected`와 상태 타입 중심으로 이관
- generated / validate / export도 새 구조 기준으로 정리

이 단계는 단순한 리팩터링이 아니라,
`서사 구조를 더 데이터적인 규칙으로 다시 설계한 작업`
이라는 점에서 시스템 기획 포인트가 크다.

---

## 11. 9단계: EditorNode 재구성

구조를 바꾼 뒤에는 편집기도 그 구조를 따라가야 했다.

그래서 EditorNode도 다시 정리했다.

핵심 변경:

- `노드 편집 / 데이터 편집` 분리
- 데이터 편집 시 더 넓은 작업 공간 사용
- 데이터 탭 분리
  - 캐릭터
  - 감정
  - 조건
  - 선택 그룹
  - 단서 분류
  - 조사
  - 질문
  - 상태
  - 룰
- 드롭다운 적극 지원
- 기본 진행 파란선 복구
- 포커스가 즉시 날아가던 데이터 편집 버그 수정

이 단계는 단순 UI 정리가 아니라,
`프로젝트가 새 구조로 실제 운영될 수 있도록 편집 환경을 다시 맞춘 작업`
이었다.

---

## 12. AI 활용 방식

이 프로젝트에서 AI는 단순한 텍스트 생성기로 쓰지 않았다.

실무적으로는 아래 역할에 집중했다.

- 데이터 구조 정리 보조
- 문체 보정
- 정합성 검수 자동화
- 반복 작업 정리
- 문서화 가속
- 에디터 / 런타임 리팩터링 보조

즉 AI를 `대사를 대신 써주는 도구`보다,
`제작 파이프라인과 시스템 운영을 돕는 도구`
로 활용했다는 점이 중요하다.

이건 포트폴리오에서 꽤 강한 포인트다.

---

## 13. 현재 프로젝트 상태

현재 `경성뎐`은 다음을 갖춘 상태다.

- 조사형 내러티브 런타임
- 스탠딩/포커스/모션/FX 연출
- 질문 해결 / 단서 연결 / 증거 제시 / 모순 판별
- 선택 이력과 조건 그룹 기반 구조
- 자체 Node Editor
- generated xlsx / validate / export 파이프라인
- 저장/이어하기 QA 자동화

즉 단순히 한 편의 ADV를 만든 프로젝트가 아니라,

`콘텐츠 구조, 데이터 스키마, 저작 툴, 검수 파이프라인, 추리 루프까지 직접 설계하고 운영한 프로젝트`

로 설명할 수 있다.

---

## 14. 포트폴리오에서 강조할 포인트

### 14-1. 시스템 기획 관점

- 내러티브를 씬/대사/선택/분기/조건 구조로 설계
- 선택 이력, 증거 보유, 상태값을 구분해 조건 판정 체계로 정리
- 추리 루프를 UI와 데이터 구조까지 포함해 구현

### 14-2. 툴/파이프라인 관점

- xlsx, game_data, generated xlsx, validate, EditorNode를 잇는 작업 구조 설계
- 레거시를 걷어내고 데이터 중심 구조로 이관
- 직접 사용하는 제작 툴을 프로젝트에 맞게 개선

### 14-3. 내러티브 게임 기획 관점

- 세계관, 캐릭터, 관계, 시대극 톤을 유지하면서도
- 플레이어가 실제로 질문을 붙들고 근거를 제출하게 만들었다

즉 포트폴리오에서는 이 프로젝트를
`분위기 좋은 비주얼노벨`로 설명하면 아깝고,

반드시

`내러티브 게임의 데이터 구조와 추리 루프를 설계하고, 운영 툴과 검수 구조까지 직접 만든 시스템 기획 프로젝트`

로 설명해야 한다.

---

## 15. 한 줄 요약

`경성뎐`은 1930년대 경성을 배경으로 한 조사형 서사 게임을 만들며, 하드코딩 프로토타입에서 시작해 데이터 구조, xlsx 파이프라인, Node Editor, 조건 중심 설계, 조사 수첩, 추리 루프, 자동 QA까지 직접 확장해 온 시스템 기획 프로젝트다.
