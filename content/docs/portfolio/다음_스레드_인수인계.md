## 2. 반드시 알고 시작할 운영 원칙

### 2-1. 현재 메인 데이터 원본

현재 실질적인 메인 원본은 `game/data/game_data.js`다.

이유:

- 노드형 에디터도 결국 JSON 데이터를 직접 다룸
- 시나리오 고속 수정은 지금 `game_data.js` 직접 편집이 가장 빠름
- `script.xlsx`는 메인 원본이라기보다 대량 수정 / 검수 / 공유용 보조 포맷

즉 현재 운영 원칙은 아래와 같다.

- 평소 작업: `game_data.js`
- 대량 편집 / 검수 필요 시: generated xlsx 생성 후 수동 복붙

### 2-2. `script.xlsx` 관련 주의

기존 `script.xlsx`를 파이썬으로 직접 안전하게 갱신하는 건 아직 비추천이다.

이유:

- Excel 표 메타데이터가 깨질 수 있음
- 실제 Excel에서 복구 경고가 뜬 적 있음

현재 안전한 흐름:

1. `game_data.js` 수정
2. `content/tools/json_to_generated_xlsx.py` 실행
3. `content/generated/script.generated.xlsx` 생성
4. 필요한 시트만 사람이 원본 `script.xlsx` 표에 복붙

## 3. 한글 깨짐 관련 주의

터미널에서 한글이 깨져 보이면 파일 손상으로 단정하지 말 것.

이 환경에선 먼저 출력 인코딩 문제를 의심해야 한다.

PowerShell에서 한글 파일 확인 시 권장 방식:

```powershell
chcp 65001 > $null
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
Get-Content '경로' -Encoding utf8
```

즉:

- 한글 깨짐 = 우선 환경 문제 의심
- 파일 손상 여부는 UTF-8 기준으로 다시 읽고 판단

## 5. 지금까지 확정된 큰 방향

### 5-1. 프로젝트 방향

이 프로젝트는 아래의 발전 흐름을 포트폴리오 핵심으로 삼는다.

1. ADV 엔진 직접 제작
2. 초기 하드코딩 프로토타입
3. 엑셀 기반 데이터 분리
4. `xlsx -> game_data` 파이프라인 구축
5. 카드형 에디터 실험
6. 노드형 에디터로 메인 툴 전환
7. generated xlsx를 보조 포맷으로 운용
8. 세계관 / 캐릭터 / 관계 설계 고도화

### 5-2. 에디터 방향

- 카드형 에디터는 사실상 폐기
- `EditorNode`가 메인 작업 툴
- `xlsx`는 대량 편집 전용 보조 포맷

### 5-3. 변수 네이밍 규칙

플래그 / 시스템 변수는 `PascalCase` 사용.

예:

- `InvestigationScore`
- `SongsoonTrust`
- `ResonanceLevel`
- `EditorRel`
- `InfoLevel`
- `EndingAScore`
- `OkryunPushed`
- `CalledEditor`

단, `SceneID` 값이나 파일명은 기존 리소스 이름 유지 가능.
