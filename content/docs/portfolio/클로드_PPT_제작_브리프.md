# 클로드 PPT 제작 브리프

## 1. 문서 목적

이 문서는 Claude가 이 프로젝트를 바탕으로 `제출용 문서형 PPT 포트폴리오`를 제작할 때 따라야 하는 기준을 정리한 브리프다.

이 작업의 핵심 전제는 아래와 같다.

- 이 결과물은 발표용 슬라이드가 아니다
- 슬라이드만 읽어도 이해되는 `문서형 포트폴리오`여야 한다
- 시각 자료가 많고, 읽기 편하며, 디자인 완성도가 높아야 한다
- 그러나 화려한 장식보다 구조적 설득력이 더 중요하다

한 줄로 정리하면 아래와 같다.

`게임기획자의 시스템 기획 프로젝트를, 시각 자료 중심의 읽는 PPT 문서로 정리하는 작업`

---

## 2. Claude가 먼저 이해해야 하는 포지셔닝

이 프로젝트는 아래처럼 보이면 안 된다.

- 게임 하나를 만든 프로젝트
- AI를 써서 빠르게 구현한 프로젝트
- 단순히 UI나 툴을 붙인 프로젝트
- 시나리오나 연출 중심 포트폴리오

반대로 아래처럼 보여야 한다.

- 시스템 기획 프로젝트
- 콘텐츠 제작 구조를 설계한 프로젝트
- 데이터 구조, 에디터, 검증 흐름을 함께 정리한 프로젝트
- 제작 과정 전체를 개선한 기획자의 작업

따라서 PPT 전반의 중심 문장은 아래 기준을 따른다.

`이 프로젝트는 게임 결과물 소개가 아니라, 게임 콘텐츠를 반복 생산하고 수정하고 검증할 수 있는 구조를 설계한 시스템 기획 프로젝트다.`

## 4. 최종 산출물 기준

Claude는 가능하면 `실제 PPT 파일`까지 만들어야 한다.

권장 산출물은 아래와 같다.

- 제출용 PPT 원본 파일
- PDF로 내보내도 문제없는 구조
- 텍스트 수정이 가능한 편집형 슬라이드

만약 현재 환경에서 직접 PPT 파일 생성이 어렵다면, 최소한 아래 산출물까지는 만들어야 한다.

- 페이지별 레이아웃이 포함된 제작 스펙 문서
- 각 페이지에 들어갈 텍스트가 정리된 문서
- 어떤 시각 자료가 필요한지 명시한 에셋 체크리스트

단, 가장 우선되는 목표는 `실제 제출 가능한 수준의 PPT`를 만드는 것이다.

## 5. 페이지 수와 구성 원칙

권장 분량은 `12페이지`다.

페이지는 아래 성격을 따른다.

1. 표지
2. 프로젝트 개요
3. 문제 정의
4. 기획 목표
5. 시스템 구조
6. 데이터 구조 정리
7. 에디터 설계
8. 작업 흐름과 검증 방식
9. Before / After
10. 결과와 개선점
11. 회고
12. 내가 보여주고 싶은 역량

이 순서는 가능하면 유지한다.

## 6. 시각 자료 중심 문서라는 점을 반드시 지킬 것

이 PPT는 글이 많은 문서가 아니라 `시각 자료가 설명을 대신하는 문서`여야 한다.

권장 비율은 아래와 같다.

- 시각 자료 60~70%
- 텍스트 30~40%

각 페이지는 가능하면 아래 구조를 따른다.

- 제목 1줄
- 설명 문단 3~5줄
- 메인 시각 자료 1개
- 하단 요약 박스 또는 강조 문장 1개

시각 자료는 아래 우선순위로 사용한다.

1. 노드 에디터 전체 화면
2. 데이터 패널 화면
3. 조건 / 상태 / 게이지 구조 캡처
4. 검증, 미리보기, 편의 기능 화면
5. 실제 게임 화면

게임 화면은 보조 증거이고, 에디터와 구조 화면은 핵심 증거다.

## 7. 디자인 방향

디자인은 단순한 기업 보고서도 아니고, 과하게 게임스러운 슬라이드도 아니어야 한다.

추천 방향은 아래와 같다.

- `에디토리얼 + 아카이브 + 문서형` 느낌
- 너무 화려하지 않지만 분명히 예쁜 구성
- 차분한 종이 톤 또는 아이보리 계열 배경
- 진한 먹색 또는 차콜 계열 텍스트
- 강조색은 짙은 적갈색, 딥 레드, 브라운 계열
- 그리드와 여백이 살아 있는 정리된 레이아웃

피해야 할 방향은 아래와 같다.

- 전형적인 스타트업 발표 슬라이드
- 지나치게 게임 UI 같은 네온 스타일
- 보라색 위주의 흔한 AI 느낌 디자인
- 블랙 배경 위 강한 형광 포인트
- 지나치게 장식적인 영화 포스터 느낌

즉, `기획 문서를 시각적으로 세련되게 정리한 편집 디자인`에 가까워야 한다.

## 8. 텍스트 톤 기준

Claude는 문장을 짧게 줄일 수는 있지만, 의미를 비우면 안 된다.

좋은 문장 기준은 아래와 같다.

- 왜 이런 판단을 했는지 설명한다
- 구조적 이유를 말한다
- 결과보다 과정과 개선을 보여준다
- 기획자의 시선을 유지한다

예시로 아래 같은 문장은 적절하다.

`콘텐츠 양이 늘어날수록 필요한 것은 더 많은 하드코딩이 아니라, 구조를 한눈에 볼 수 있는 편집 도구였다.`

`이 프로젝트에서는 기획 개념이 실제 데이터 구조에 그대로 이어지도록 Condition 중심 구조로 재정리했다.`

`AI는 결과물을 대신 만든 것이 아니라 반복 구현 속도를 높여 기획 검증 시간을 확보하는 도구로 사용했다.`

반대로 아래 표현은 피한다.

- 열심히 만들었다
- 많은 기능을 구현했다
- 다양한 기술을 사용했다
- AI를 활용해 빠르게 만들었다
- 게임을 하나 완성했다

## 9. 절대 하지 말아야 할 것

- 프로젝트를 시나리오 포트폴리오처럼 보이게 만들지 말 것
- 게임 플레이 장면만 크게 쓰고 구조 설명을 약하게 만들지 말 것
- 텍스트를 너무 줄여서 읽는 문서의 기능을 잃지 말 것
- 반대로 글만 많고 시각 자료가 약한 문서를 만들지 말 것
- 문서에 없는 기능이나 성과를 지어내지 말 것
- AI 활용을 프로젝트의 핵심처럼 과장하지 말 것

## 10. 이미지가 부족할 때의 처리 원칙

필요한 스크린샷이나 시각 자료가 부족하면 Claude는 임의 이미지를 생성해서 끼워 넣지 말고, 아래처럼 명확한 자리 표시를 해야 한다.

- `[스크린샷 필요: 노드 에디터 메인 화면]`
- `[비교 이미지 필요: Before / After 데이터 구조 표]`
- `[다이어그램 필요: 편집 -> 변환 -> 검증 -> 반영 흐름]`

즉, 없는 증거를 꾸며 넣지 말고 `무엇이 더 필요했는지`를 정확하게 표시해야 한다.

## 11. Claude의 작업 순서

Claude는 아래 순서로 작업하는 것이 가장 좋다.

1. 필수 문서 3개를 먼저 읽고 구조를 이해한다
2. 보조 문서로 표현과 맥락을 보강한다
3. 12페이지 기준 목차를 확정한다
4. 페이지별 텍스트를 정리한다
5. 페이지별 시각 자료 배치를 설계한다
6. 전체 디자인 톤을 통일한다
7. 실제 PPT 파일로 만든다
8. PDF로 읽을 때도 자연스러운지 최종 점검한다

---

## 12. Claude에게 기대하는 결과물 품질

최종 결과물은 아래 조건을 만족해야 한다.

- 포트폴리오처럼 읽히고
- PPT처럼 깔끔하게 정리되어 있으며
- PDF로 제출해도 완성도가 높고
- 게임기획자의 시스템 설계 역량이 분명히 보이는 결과물

즉, 예쁜 것만으로는 부족하고 `읽었을 때 설득되는 문서`여야 한다.

---

## 13. Claude에게 바로 줄 수 있는 요청 핵심

Claude에게 요청할 때는 아래 핵심을 반드시 포함한다.

- 이건 발표용이 아니라 문서형 PPT다
- 게임기획자, 그중에서도 시스템 기획 포트폴리오다
- 시각 자료를 많이 써야 한다
- 에디터 / 구조 / 검증이 핵심 증거다
- 기존 문서를 읽고 그 기준을 유지해야 한다
- 문서에 없는 내용을 지어내면 안 된다

---

## 14. 마지막 체크리스트

Claude가 작업을 끝냈을 때 아래 항목을 확인해야 한다.

- 각 페이지가 슬라이드만 읽어도 이해되는가
- 텍스트와 시각 자료 비율이 적절한가
- 디자인이 과하거나 흔하지 않은가
- 시스템 기획 역량이 분명하게 드러나는가
- 에디터와 구조 화면이 충분히 중심에 오는가
- 회고와 판단이 살아 있는가
- 발표용 PPT처럼 과하게 비어 있지 않은가
