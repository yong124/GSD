# 제출용 포트폴리오 페이지별 문구 초안

이번 문구는 발표 멘트용이 아니라 `페이지 자체만 읽어도 이해되는 문서형 설명`을 기준으로 작성한다.

## Page 1. 표지

### 제목

`콘텐츠 제작 구조까지 설계한 ADV 시스템 기획 프로젝트`

### 부제

`시스템 구조 정리 / 데이터 설계 / 노드 에디터 제작 / 검증 흐름 구축`

### 본문

브라우저 기반 ADV 프로젝트를 직접 설계하면서, 단순히 게임 결과물을 만드는 데 그치지 않고 콘텐츠 제작 구조와 데이터 시스템, 노드 에디터, 검증 흐름까지 함께 정리한 작업이다.

### 하단 요약 문장

`게임 하나를 만든 것이 아니라, 게임을 계속 만들 수 있는 구조를 설계한 프로젝트`

---

## Page 2. 프로젝트 개요

### 제목

`프로젝트 개요`

### 본문

이 프로젝트는 브라우저 기반 ADV 게임으로 시작했다. 초기에는 하드코딩 중심으로 빠르게 구조를 검증했지만, 콘텐츠 양이 늘어나면서 데이터 구조 분리와 편집 환경 정리가 필요해졌다. 이후 시스템 개념을 재정리하고, 노드 에디터와 검증 흐름까지 확장하면서 제작 환경 전체를 정리하는 프로젝트로 발전했다.

### 요약 박스

- 초기: 하드코딩 기반 프로토타입
- 확장: 데이터 구조 분리
- 현재: 에디터와 검증 흐름까지 포함한 제작 환경

---

## Page 3. 문제 정의

### 제목

`기존 방식의 문제`

### 본문

콘텐츠가 적을 때는 하드코딩 구조로도 빠르게 작업할 수 있었지만, 분기와 상태 변화가 늘어나자 수정 비용이 빠르게 증가했다. 분기 구조가 코드 안에 퍼져 있어 기획 검토가 어려웠고, 작업자가 직접 콘텐츠를 수정하기도 불편했다. 결국 문제는 기능 구현 자체보다, 콘텐츠를 운영 가능한 구조로 다루기 어렵다는 점에 있었다.

### 하단 요약 문장

`문제는 게임 구현이 아니라, 콘텐츠를 반복 수정하고 검증할 수 있는 구조의 부재였다.`

---

## Page 4. 기획 목표

### 제목

`어떤 방향으로 해결하려 했는가`

### 본문

이 프로젝트에서 중요했던 것은 단순히 편집 기능을 추가하는 것이 아니었다. 콘텐츠를 코드에서 분리해 데이터 단위로 관리하고, 기획 개념이 실제 데이터 구조에 자연스럽게 이어지도록 정리해야 했다. 또한 분기 구조는 시각적으로 다뤄야 했고, 편집 이후 검증과 실제 반영까지 일관된 흐름이 필요했다.

### 포인트 정리

- 콘텐츠를 데이터 중심으로 관리
- 기획 개념을 일관된 구조로 정리
- 분기 흐름을 시각적으로 편집
- 검증과 반영 흐름까지 연결

---

## Page 5. 시스템 구조

### 제목

`시스템 구조 정리`

### 본문

전체 구조는 질문, 선택지, 결과 흐름을 중심으로 구성하고, 분기 조건은 Condition 개념으로 통합했다. 여기에 상태값, 게이지, 조사와 증거, 인물 상태 같은 요소들을 연결해 실제 플레이와 기획 판단이 데이터 구조 안에서 이어지도록 설계했다. 이 과정에서 중요한 기준은 기능을 늘리는 것이 아니라, 구조를 이해하기 쉽게 만드는 것이었다.

### 하단 요약 문장

`기획 개념이 실제 데이터 구조로 이어지도록 정리한 것이 이 프로젝트의 핵심 설계 포인트였다.`

---

## Page 6. 데이터 구조 정리

### 제목

`레거시 구조를 어떻게 정리했는가`

### 본문

초기 구조에는 Flag, Rule 같은 레거시 개념이 남아 있었고, 실제 운영 과정에서는 의미가 겹치거나 역할이 모호한 부분이 존재했다. 따라서 필요한 데이터는 남기되 개념이 중복되거나 운용 기준이 불명확한 구조는 정리하고, Condition 중심 구조로 통합했다. 이 정리는 단순한 이름 변경이 아니라, 실제 기획과 운영 기준에 맞게 구조를 재분류한 과정이었다.

### 비교 포인트

- 남길 데이터와 지울 레거시를 구분
- 분기 판단 기준을 Condition으로 통합
- 상태, 게이지, 조사 구조를 역할별로 재정리

---

## Page 7. 에디터 설계

### 제목

`노드 에디터를 설계한 이유`

### 본문

콘텐츠 양이 늘어날수록 필요한 것은 결과 화면보다 편집 도구였다. 분기 흐름을 한눈에 보고 수정 지점을 빠르게 찾으려면 시각적 편집 환경이 필요했고, 데이터 패널과 흐름 편집을 함께 연결해야 했다. 그래서 이 프로젝트에서는 노드 기반 에디터를 중심으로, 실제 작업자가 직접 콘텐츠를 수정하고 확인할 수 있는 구조를 설계했다.

### 하단 요약 문장

`에디터는 편의 기능이 아니라, 콘텐츠 생산 구조를 지탱하는 핵심 도구였다.`

---

## Page 8. 작업 흐름과 검증 방식

### 제목

`편집 이후의 흐름까지 설계하기`

### 본문

에디터를 만드는 것만으로는 충분하지 않았다. 편집한 데이터가 실제 게임 구조로 안전하게 변환되고, 누락값이나 잘못된 연결이 검증 단계에서 확인되며, 최종적으로 게임에 반영되는 흐름까지 함께 설계해야 했다. 또한 FK와 Enum 기반 데이터는 선택형 UI로 연결하고, 이미지 미리보기와 빠른 확인 기능을 붙여 실제 작업 흐름을 개선했다.

### 포인트 정리

- 편집 -> 변환 -> 검증 -> 반영
- FK / Enum 기반 선택 UI
- 이미지 미리보기와 확인 기능
- 잘못된 데이터의 조기 발견

---

## Page 9. Before / After

### 제목

`프로젝트가 어떻게 발전했는가`

### 본문

이 프로젝트는 처음부터 완성형 구조로 시작한 것이 아니라, 실제 문제를 만나며 단계적으로 개선됐다. 초기에는 하드코딩 중심 프로토타입이었고, 이후 데이터 구조를 분리하고, 레거시 개념을 정리해 Condition 중심 구조로 통합했다. 현재는 에디터, 검증, 미리보기, 실제 반영 흐름까지 연결된 형태로 발전했다.

### 하단 요약 문장

`이 프로젝트의 강점은 결과물 하나보다, 제작 구조를 계속 개선해 온 과정 자체에 있다.`

---

## Page 10. 결과와 개선점

### 제목

`무엇이 달라졌는가`

### 본문

구조가 정리된 이후에는 콘텐츠 수정 속도와 검토 속도가 모두 빨라졌다. 분기와 상태 구조를 더 명확하게 이해할 수 있게 되었고, 반복 수정 비용도 줄었다. 무엇보다 중요한 변화는, 이후 콘텐츠를 더 안정적으로 생산하고 확장할 수 있는 기반이 생겼다는 점이다.

### 요약 박스

- 수정 속도 향상
- 구조 이해도 향상
- 반복 수정 비용 감소
- 확장 기반 확보

---

## Page 11. 회고

### 제목

`회고`

### 본문

이번 프로젝트를 통해 가장 크게 확인한 것은, 콘텐츠 중심 프로젝트일수록 결과 화면만큼이나 제작 구조가 중요하다는 점이었다. 빠르게 검증하기 위해 하드코딩으로 시작한 판단은 유효했지만, 이후 구조를 제때 정리하지 않으면 유지 비용이 커진다는 점도 분명히 드러났다. 반대로 시스템 개념, 데이터 구조, 에디터, 검증 흐름을 함께 설계한 경험은 이후 프로젝트에서도 큰 자산이 될 수 있다고 느꼈다.

### 회고 포인트

- 빠른 검증과 구조 정리의 균형이 중요했다
- 기획 개념은 결국 데이터 구조로 이어져야 했다
- 에디터는 제작 효율을 크게 좌우했다

---

## Page 12. 내가 보여주고 싶은 역량

### 제목

`이 프로젝트를 통해 보여주고 싶은 역량`

### 본문

이 프로젝트를 통해 보여주고 싶은 역량은 단순한 기능 기획 능력이 아니다. 나는 시스템을 구조화해서 정의하고, 콘텐츠 생산 흐름을 기획 관점에서 정리하며, 에디터와 데이터 구조를 함께 보는 방식에 강점을 갖고 있다. 또한 AI를 결과 대체가 아니라 반복 구현 가속 도구로 활용해, 실제 기획 판단과 검증에 더 많은 시간을 투입하는 작업 방식을 지향한다.

### 마무리 문장

`콘텐츠를 기획하는 데서 멈추지 않고, 그 콘텐츠가 생산되고 수정되고 검증될 수 있는 구조까지 함께 설계하는 기획자`

## 2. 문서형 포트폴리오에서의 문장 기준

- 문장은 길지 않지만 의미는 분명해야 한다
- 결과보다 판단과 구조를 설명해야 한다
- 한 페이지에는 하나의 메시지만 두는 것이 좋다
- 시각 자료가 설명을 대신할 수 있게 문장은 보조 역할을 해야 한다
