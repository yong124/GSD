# 경성뎐 최종 테이블 구조 정리

이 문서는 현재 경성뎐에서 합의한 **최종 데이터 구조 방향**을 정리한 문서다.

기준은 아래와 같다.

- 조건 판정은 `ConditionTable` 중심으로 처리한다.
- 선택 이력은 런타임에 저장하고, `ConditionType = ChoiceSelected`로 읽는다.
- 증거 보유 여부는 런타임에 저장하고, `ConditionType = EvidenceOwned`로 읽는다.
- 신뢰도/공명도/조사도/의식 이해도 같은 실제 상태값은 런타임 상태로 저장하고 조건에서 읽는다.
- `Question / Rule / StateDescriptor`는 현재 라운드에서 **테이블화 보류**이며, 런타임 하드코딩 유지다.

---

# 1. 구조 원칙

## 1-1. 무엇을 테이블로 두는가

- 씬 구조
- 대사 구조
- 선택지 구조
- 분기 구조
- 증거 구조
- 캐릭터 구조
- 감정 이미지 구조
- 조사 구조
- 조건 구조

## 1-2. 무엇을 런타임 상태로 두는가

- 선택 이력
- 증거 보유 여부
- 신뢰도
- 공명도
- 조사 점수
- 의식 이해도

## 1-3. 이번 문서에서 보류하는 것

- `Question`
- `Rule`
- `StateDescriptor`

이 셋은 아직 테이블로 확정하지 않고, 현재 게임 런타임 하드코딩 구조를 유지한다.

---

# 2. ConditionTable

게임 내 표시 조건, 분기 조건, 선택지 노출 조건을 정의한다.

같은 `ConditionGroupID`를 가진 행은 **AND 조건 묶음**으로 처리한다.

| 컬럼 | 설명 | 참조 |
| --- | --- | --- |
| `ConditionID` | 조건 고유 ID | - |
| `ConditionGroupID` | 동일 조건 묶음 ID | - |
| `ConditionType` | 검사 타입 | `EConditionType` |
| `ConditionTargetID` | 검사 대상 ID | 타입별 참조 |
| `CompareType` | 비교 방식 | `ECompareType` |
| `ConditionValue` | 비교값 | - |

## EConditionType

- `Trust`
- `EvidenceOwned`
- `ChoiceSelected`
- `RevealedCharacter`
- `SceneProgressIndex`
- `StateValue`

## ConditionTargetID 해석

| ConditionType | ConditionTargetID 의미 |
| --- | --- |
| `Trust` | `CharacterTable.CharacterID` |
| `EvidenceOwned` | `EvidenceTable.EvidenceID` |
| `ChoiceSelected` | `ChoiceTable.ChoiceID` |
| `RevealedCharacter` | `CharacterTable.CharacterID` |
| `SceneProgressIndex` | 비움 또는 `0` |
| `StateValue` | `EStateType` 값 |

## 메모

- `SongsoonTrust`는 `ConditionType = Trust`, `ConditionTargetID = Songsoon`으로 처리한다.
- `ReadRitualScore`, `ResonanceLevel`, `InvestigationScore`는 `ConditionType = StateValue`로 처리한다.
- 선택 이력은 `ChoiceSelected`로 읽는다.

---

# 3. EStateType

현재 게임에서 별도 상태값으로 유지하는 항목이다.

- `ReadRitualScore`
- `ResonanceLevel`
- `InvestigationScore`

## 메모

- `SongsoonTrust`는 별도 `EStateType`이 아니라 `Trust(Songsoon)`로 처리한다.
- 나중에 상태값이 늘어나면 이 enum만 확장하면 된다.

# 7. ChoiceTable

플레이어가 실제로 고르는 개별 선택지다.

선택지는 **노출 조건**, **이동**, **획득 증거**, **신뢰 변화**, **공명 변화**, **상태 변화**를 가진다.

| 컬럼 | 설명 | 참조 |
| --- | --- | --- |
| `ChoiceID` | 선택지 고유 ID | - |
| `ChoiceGroupID` | 소속 그룹 ID | `ChoiceGroupTable.ChoiceGroupID` |
| `Order` | 그룹 내 출력 순서 | - |
| `Text` | 선택지 문구 | - |
| `ConditionGroupID` | 선택지 노출 조건 | `ConditionTable.ConditionGroupID` |
| `NextType` | 선택 후 이동 타입 | `ENextType` |
| `NextID` | 이동 대상 ID | Scene 또는 Dialog |
| `EvidenceID` | 획득 증거 ID | `EvidenceTable.EvidenceID` |
| `TrustCharacterID` | 신뢰도 대상 캐릭터 ID | `CharacterTable.CharacterID` |
| `TrustValue` | 신뢰도 변화값 | - |
| `ResonanceValue` | 공명도 변화값 | - |
| `StateType` | 상태 변경 타입 | `EStateType` |
| `StateValue` | 상태 변경 값 | - |

## ENextType

- `Scene`
- `Dialog`
- `None`

## NextID 해석

| NextType | NextID 의미 |
| --- | --- |
| `Scene` | `SceneTable.SceneID` |
| `Dialog` | `DialogTable.DialogID` |
| `None` | 비움 또는 `0` |

## 메모

- 선택 이력은 런타임에서 `ChoiceID` 기준으로 저장한다.
- `flag_key`, `flag_value`, `extra_flags`, `next_scene`, `next_dialogue`는 사용하지 않는다.

---

# 8. BranchTable

씬 종료 후 자동으로 평가되는 분기 테이블이다.

같은 `SceneID` 안에서 `Order` 순으로 평가하고, 처음 만족한 분기로 이동한다.

| 컬럼 | 설명 | 참조 |
| --- | --- | --- |
| `BranchID` | 분기 고유 ID | - |
| `SceneID` | 소속 씬 ID | `SceneTable.SceneID` |
| `Order` | 평가 순서 | - |
| `ConditionGroupID` | 분기 조건 | `ConditionTable.ConditionGroupID` |
| `NextScene` | 이동 대상 씬 ID | `SceneTable.SceneID` |

## 메모

- 조건이 비어 있는 기본 브랜치 1개를 두면 기본 진행선을 표현할 수 있다.
- `flag_key / flag_value` 기반 분기는 사용하지 않는다.

---

# 14. 런타임 상태와 테이블의 관계

## 14-1. 런타임에 저장하는 것

- `chosenChoiceIds`
- `ownedEvidenceIds`
- 캐릭터 신뢰도 값
- 공명도
- 조사 점수
- 의식 이해도

## 14-2. 테이블이 하는 일

- `ConditionTable`이 런타임 상태를 읽는다.
- `ChoiceTable`이 런타임 상태를 바꾼다.
- `BranchTable`이 조건 결과에 따라 씬 이동을 결정한다.

즉:

- 상태 저장 = 런타임
- 조건 정의 = 테이블
- 선택 결과 정의 = 테이블

---

# 15. 이번 구조에서 일부러 보류한 것

아래 3개는 현재 라운드에서 테이블화하지 않는다.

- `Question`
- `Rule`
- `StateDescriptor`

이유:

- 현재 질문/추리 루프가 아직 구조적으로 고정되지 않았다.
- 지금은 런타임 하드코딩이 더 빠르고 안전하다.
- 나중에 추리 UI와 질문 시스템이 확정되면 별도 테이블로 다시 설계한다.

---

# 16. 최종 요약

현재 기준 최종 구조는 아래 축으로 이해하면 된다.

- `SceneTable`
- `DialogTable`
- `ChoiceGroupTable`
- `ChoiceTable`
- `BranchTable`
- `ConditionTable`
- `EvidenceCategoryTable`
- `EvidenceTable`
- `CharacterTable`
- `CharacterEmotionTable`
- `InvestigationTable`

그리고 별도 개념은 다음처럼 본다.

- 선택 이력 = `ChoiceSelected`
- 증거 보유 = `EvidenceOwned`
- 신뢰도 = `Trust`
- 공명/조사/의식 이해도 = `StateValue`

즉 최종 방향은:

**플래그 중심 구조가 아니라, 조건 테이블이 런타임 상태를 읽고 선택 테이블이 상태를 바꾸는 구조**다.
